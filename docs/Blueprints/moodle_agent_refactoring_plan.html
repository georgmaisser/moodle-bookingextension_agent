Observation Builder: Precondition ex > 0 erzwingen
Kein Planner-Call ohne abgeschlossene Execution im selben Run

Problem
Entry 547 hat ob=1 und ex=0. Der Observation Builder hat gefeuert, obwohl kein Queue-Item je in running → succeeded/failed übergegangen ist. Das ist kein echter Observation-Trigger, sondern ein Scheduling-Artefakt.
Fix
Vor jedem Observation-Builder-Aufruf: Guard-Check auf Queue-Transition-Event. Nur wenn mindestens ein Item im aktuellen run_id einen Terminal- oder Retry-Status erreicht hat, darf der Observation Builder den Planner triggern. Kein Transition-Event → kein Planner-Call, kein Log-Eintrag mit ac=wpl.

2
st=sr-Pfad: Queue-Item-ID als Gating-Bedingung
Scheduled Re-Run darf nur mit konkretem Queue-Kontext starten
›
Problem
Entry 547 hat st=sr (Scheduled Re-Run). Er startet ohne erkennbaren Queue-Item-Kontext und produziert denselben Output wie Entry 546. Ein Re-Run ohne Queue-Kontext ist eine Race-Condition: er kann nicht wissen, ob das, worauf er reagiert, noch relevant ist.
Fix
Der st=sr-Eintrittspunkt muss zwingend eine queue_item_id oder ein confirmation_bundle_id mitführen. Ohne diesen Parameter: sofortiger Abort, kein Planner-Call, kein Log-Eintrag der Art Entry 547.
Zielbild-Bezug
Entspricht dem Prinzip: Queue ist Single Source of Truth. Kein Pfad darf am Queue-Kontext vorbei in den Agentic Loop eintreten.
3
Budget-Guard vor jedem Planner-Re-Entry
Endlosloop-Schutz: max. Planner-Calls pro Thread und Run
›
Problem
Der Loop NP → Queue-Ingestion → Planner hat keine Abbruchbedingung außer "goal reached". Im Log sieht man zwei identische Planner-Calls ohne Fortschritt. Ohne Budget-Guard ist das ein potenzieller Endlosloop.
Fix
Counter planner_call_count pro run_id persistieren. Vor jedem Planner-Dispatch: wenn planner_call_count ≥ max_planner_calls (Empfehlung: 5), dann Loop-Terminal mit strukturiertem issue_code: budget_exceeded. Dieser Code muss im Observation Builder als hard stop interpretiert werden, nicht als recoverable error.

Alle 7 Commands als vollwertige Queue-Items, nicht nur die erste Mutation
›
Problem

Der Embedding-Kontext (em=applied/off) wird heute pro Call neu entschieden, nicht am run_id oder thread_id persistiert. Entry 545 hat das Embedding berechnet; Entry 547 ignoriert es.

8
soft_block durch Layer 3 routen
Queue-State-Mapping an einer einzigen Stelle
›
Problem
Im aktuellen Flowchart verlässt soft_block aus Layer 2 den Preflight-Pfad direkt. Layer 3 hat keine Sicht auf soft_block-Entscheidungen. Audit-Logging und Queue-State-Mapping sind damit verteilt.
Fix
Alle Preflight-Entscheidungen — auch soft_block — laufen durch Layer 3. Layer 3 entscheidet ausschließlich über Queue-State-Mapping und schreibt den Audit-Event. Das bedeutet: Layer 3 bekommt den Layer-2-Output als Input und mappt ihn deterministisch auf Queue-Status.

›
Problem

10
used_triggers aus dem Response-Schema entfernen
Trigger-Klassifikation gehört in den Planner-Prompt-Kontext, nicht in den Output
›
Problem
used_triggers: ["booking.create_booking_request", "booking.skip_location_specification"] sind Task-Namen als Strings im Planner-Output. Wenn die Runtime diese jemals als Steuerlogik liest, ist das ein Freitext-Nebenkanal. Selbst wenn sie nur geloggt werden: sie verschmutzen das Response-Schema mit Debug-Information.

11
next_step_intent durch strukturierten Typ ersetzen
Was der Planner als "nächsten Schritt" meint, muss ein issue_code sein
›
Problem
Entry 547 hat next_step_intent: "Bestätigung der 7 Lectures für nächste Woche einholen." — das ist ein Freitext-String, den der Observation Builder vermutlich auswertet, um zu entscheiden, ob ein neuer Planner-Call nötig ist.
Fix
Wenn der Planner signalisieren will, dass eine Confirmation noch aussteht: strukturiertes Feld next_action: "await_confirmation" mit enum-Werten. Mögliche Werte: await_confirmation | execute | repair | done. Die Runtime mappt auf Queue-Status, nicht auf Freitext-Parsing.
Zielbild-Bezug
Leitprinzip 3: "Steuerlogik ist sprachagnostisch und basiert auf strukturierten Signalen, nicht auf Freitext." next_step_intent ist das Gegenteil davon.
